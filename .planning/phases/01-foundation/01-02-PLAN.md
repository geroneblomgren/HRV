  <name>Task 1: Create BLEService with GATT connection, 0x2A37 parsing, artifact rejection, and reconnect</name>
  <files>js/ble.js</files>
  <action>
Create js/ble.js implementing the full BLE pipeline. Import AppState and subscribe from js/state.js. Import setSetting from js/storage.js.

**Connection (BLE-01):**
- initiateConnection(): Called from Connect button click handler. Calls navigator.bluetooth.requestDevice({ filters: [{ services: ['heart_rate'] }] }). Stores device reference. Saves device name to AppState.savedDeviceName and to IndexedDB via setSetting('deviceName', name). Then calls internal connect().
- tryQuickConnect(): Checks typeof navigator.bluetooth.getDevices. If available, attempts to find device matching AppState.savedDeviceName. If found, uses it directly. Otherwise falls back to initiateConnection(). Per RESEARCH.md Pitfall 4, include typeof guard.
- disconnect(): Disconnects GATT server if connected. Sets AppState.connectionStatus = 'disconnected'.

**Internal connect() (BLE-01, BLE-04):**
- Sets AppState.connectionStatus = 'connecting'
- Uses connectWithTimeout (Promise.race with 12s timeout) per RESEARCH.md Pattern 3
- Gets primary service 'heart_rate', gets characteristic 'heart_rate_measurement'
- Calls startNotifications(), adds characteristicvaluechanged listener
- Adds gattserverdisconnected listener for auto-reconnect
- On success: resets reconnect attempt counter, sets connected=true, connectionStatus='connected', lastConnectTime=Date.now()
- On failure: sets connectionStatus='disconnected', throws error

**Reconnect (BLE-04):**
- RECONNECT_TIMEOUTS = [1000, 2000, 4000, 8000, 8000] (5 attempts per user decision)
- onDisconnected(): sets connected=false, connectionStatus='reconnecting', calls scheduleReconnect()
- scheduleReconnect(): if attempts exhausted, sets connectionStatus='disconnected' and showManualReconnect=true. Otherwise waits the delay then calls connect(). On connect failure, calls scheduleReconnect() again.
- After successful reconnect, re-registers startNotifications and listener (per RESEARCH.md anti-pattern note)

**0x2A37 Parsing (BLE-02):**
- parseHRMNotification(event): Exactly per RESEARCH.md Pattern 2. Parse flags byte, determine HR format (UINT8/UINT16), check Energy Expended present, check RR Present. Extract HR value. Loop remaining bytes as UINT16 LE pairs, convert from 1/1024 sec to ms: (rawRR / 1024) * 1000. Return { bpm, rrValues }.
- CRITICAL: Must iterate ALL remaining byte pairs, not just the first. This is RESEARCH.md Pitfall 2.

**Artifact Rejection (BLE-05):**
- Two-tier rejection per RESEARCH.md Pattern 4:
  - Tier 1: Absolute bounds — reject if ms < 300 or ms > 2000
  - Tier 2: Relative bounds — if history has >= 5 values, compute running median of last 5 clean RR values, reject if deviation > 20%
- On rejection: increment AppState.artifactCount, interpolate with last clean value (do NOT delete — RESEARCH.md Pitfall 3)
- On acceptance: push to _rrHistory (keep max 20), increment AppState.rrCount, compute AppState.currentHR = Math.round(60000 / ms)
- writeToCircularBuffer(ms): Write to AppState.rrBuffer at AppState.rrHead, increment rrHead modulo 512

**Notification handler:**
- handleNotification(event): calls parseHRMNotification, updates AppState.currentHR with bpm, calls ingestRRValues with the rrValues array

**Exports:** initiateConnection, tryQuickConnect, disconnect
