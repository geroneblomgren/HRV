  <action>
Create js/tuning.js as an ES module following the session controller pattern from practice.js.

**Imports:**
- `{ AppState }` from './state.js'
- `{ initAudio, startPacer, stopPacer }` from './audio.js'
- `{ initDSP, tick, computeSpectralRSA }` from './dsp.js'

**Exported functions:**

1. `startTuning(savedFreqHz)` — Entry point called from practice.js before session starts.
   - Takes the user's stored resonance frequency in Hz (e.g., 0.0833 for 5.0 BPM)
   - Generates 5 candidate frequencies: stored freq ± 0.5 BPM in 0.25 BPM steps
     - Example: stored = 5.0 BPM → candidates = [4.5, 4.75, 5.0, 5.25, 5.5] BPM
     - Convert each to Hz: bpm / 60
   - If savedFreqHz is null (first session, no Discovery result), use extended range: [4.5, 5.0, 5.5, 6.0, 6.5] BPM (the standard Discovery frequencies)
   - Sets AppState.tuningPhase = 'scanning'
   - Sets AppState.tuningStoredFreqBPM = savedFreqHz * 60 (or 0 if null)
   - Sets AppState.tuningCandidateCount = 5
   - Sets AppState.tuningResults = []
   - Calls initAudio() and initDSP()
   - Starts the first candidate block
   - Returns a Promise that resolves with `{ freqHz, freqBPM, rsaAmplitude }` when tuning completes

2. `stopTuning()` — Force-stop tuning (e.g., if user cancels). Cleans up timers and pacer.

3. `getTuningResult()` — Returns the current tuning result object or null if not complete.

**Internal logic — candidate block cycling:**

Each candidate gets 12 seconds of guided breathing (60s total / 5 candidates):

- `_startCandidate(index)`:
  - Set AppState.tuningCandidateIndex = index
  - Set AppState.tuningCurrentFreqBPM = candidates[index] BPM
  - Call startPacer(candidates[index] Hz) to start bowl echoes at this rate
  - Start a 1-second DSP tick interval (same pattern as practice.js)
  - After 12 seconds, call `_endCandidate(index)` via setTimeout

- `_endCandidate(index)`:
  - Call stopPacer()
  - Clear DSP tick interval
  - Measure RSA: call `computeSpectralRSA(12, candidateFreqHz)` — 12-second window at the candidate frequency
  - Store result: push `{ freqBPM, rsaAmplitude }` to AppState.tuningResults
  - Update AppState.tuningProgress = (index + 1) / candidateCount
  - If more candidates remain: start next with 0ms gap (immediate transition, bowl pacer provides natural pacing)
  - If all candidates done: call `_selectWinner()`

- `_selectWinner()`:
  - Find candidate with highest rsaAmplitude from tuningResults
  - Set AppState.tuningSelectedFreqBPM = winner BPM
  - Set AppState.tuningSelectedRSA = winner RSA amplitude
  - Set AppState.tuningPhase = 'result'
  - Resolve the promise with `{ freqHz: winnerBPM / 60, freqBPM: winnerBPM, rsaAmplitude: winnerRSA }`

**Important implementation details:**
- The DSP tick during tuning calls `tick(elapsedSeconds)` where elapsedSeconds is time since tuning started (not session start). This keeps the DSP pipeline alive for RSA computation.
- However, the calibration gate in dsp.js requires 120s. For tuning, we bypass this by using `computeSpectralRSA()` directly, which has its own RR buffer read logic independent of the tick calibration gate. The tick calls are still needed to keep AppState.coherenceScore updating (even though we don't display it during tuning).
- Actually, `computeSpectralRSA()` reads directly from `AppState.rrBuffer` — it does NOT depend on tick(). So for tuning, we just need the RR data to be flowing. The pacer drives breathing, the HRM captures RR intervals, and computeSpectralRSA reads the buffer. We DO still want tick() running so spectralBuffer stays current for any renderers, but the tuning measurement itself is via computeSpectralRSA.
- Use a module-level `_resolve` reference to resolve the Promise returned by startTuning.
