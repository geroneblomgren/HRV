1. Open the app and open DevTools console
2. Verify no JS errors on load

**Live session test:**
1. Connect HRM and start a practice session
2. After tuning completes, verify:
   - BPM badge appears inside the pacer circle showing the tuned frequency (e.g., "5.0")

   - `AppState.paceControllerActive` remains `false` (calibration gate)
   - `AppState.pacingFreq` does not change from tuned value
4. After 120 seconds, observe:
   - If phase lock drops below 50, watch `AppState.pacingFreq` in console — it should remain stable for 10 seconds, then begin tiny increments (~0.00033 Hz per second)
   - BPM badge updates to reflect the new frequency
   - Bowl echo spacing shifts gradually — no click, pop, or abrupt change
5. Verify hard bound:
   - `AppState.pacingFreq` never exceeds `AppState.pacingFreqTuned ± 0.5/60`
   - If clamped at bound, badge turns amber (#f59e0b)
6. When phase lock recovers above 50:
   - `AppState.paceControllerActive` becomes `false`
   - Frequency stays at whatever value achieved lock (does NOT creep back to tuned)
7. End the session and verify summary:
   - If pace drifted, "Pace: X.X → X.X BPM" text appears
   - If pace did not drift, text is hidden
